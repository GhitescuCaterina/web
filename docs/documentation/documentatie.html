1. Introduction

1.1 Purpose

vroomData is a web-based tool designed to facilitate the visualization and multi-criteria comparison of public data regarding the auto park in Romania over the past ten years. By utilizing a proprietary REST/GraphQL API, the application aims to provide users with comprehensive and interactive statistics, visualizations, and geographic representations of the automotive data. Users can export the generated statistics and visualizations in various formats, including CSV, WebP, and SVG.

1.2 Intended Audience

The primary users of the APa (Auto Park Smart Explorer) application include:

  • Data Analysts: Professionals who analyze automotive data to identify trends, patterns, and insights over the last decade.
  • Government Officials: Authorities and policymakers who need to understand the dynamics of the auto park in Romania to make informed decisions.
  • Researchers and Academics: Scholars conducting studies on automotive trends, environmental impact, and transportation systems.
  • General Public: Individuals interested in exploring and understanding the evolution of the automotive landscape in Romania.
  • Business Analysts: Market analysts and strategists in the automotive industry who need data-driven insights for business planning and decision-making.

1.3 Product Scope

"vroomData (Auto Park Smart Explorer)" is a project aimed at providing a thorough analysis of Romania's automotive landscape over the past decade. It utilizes a proprietary REST/GraphQL API to offer users a platform for exploring and comparing various aspects of the country's auto park statistics.

Through vroomData, users can access and visualize data on vehicle registrations, types of vehicles, fuel types, and ownership trends. The platform offers intuitive interfaces and advanced filtering options for easy navigation and extraction of insights.

vroomData generates diverse visualizations, including graphical representations and cartographic visualizations using specialized web services. Users can export statistics and visualizations in CSV, WebP, and SVG formats for further analysis or integration into reports and presentations.

Overall, vroomData serves as a valuable resource for researchers, policymakers, industry professionals, and enthusiasts interested in understanding the dynamics of Romania's auto industry.

2. Overall Description

2.1 Product Perspective

vroomData is envisioned as a sophisticated, user-friendly web application that leverages modern web technologies to offer comprehensive analysis and visualization of Romania's automotive data. The product is designed to cater to various stakeholders, including data analysts, government officials, researchers, and the general public. It provides a seamless interface for accessing, exploring, and exporting detailed automotive statistics and trends over the past decade. The application integrates multiple data visualization techniques and geographic mapping services to present data in an insightful and engaging manner. Users can interact with the data through a responsive web interface, making it accessible on various devices, including desktops, tablets, and smartphones.

2.2 Product Functions

Here's a list of major functions in the implementation of the "vroomData (Auto Park Smart Explorer)" project:

  • Data Retrieval: Function to fetch public data related to Romania's auto park from the REST/GraphQL API.
  • Data Processing: Function to process the retrieved data, including cleaning, formatting, and organizing it for analysis.
  • Filtering: Functionality to allow users to filter data based on various criteria such as vehicle type, fuel type, region, and time period.
  • Statistical Analysis: Functions to calculate and present statistical metrics such as averages, totals, trends, and distributions.
  • Visualization Generation: Functions to generate graphical representations of data, including charts, graphs, and histograms to visually convey insights.
  • Cartographic Visualization: Functions to integrate with specialized web services for generating cartographic visualizations, such as maps showing spatial distribution of vehicle registrations.
  • Exporting: Functions to enable users to export data, statistics, and visualizations in formats like CSV, WebP, and SVG for further analysis or integration into external documents.
  • User Authentication and Authorization: Functions to manage user authentication and authorization, ensuring secure access to the platform and its functionalities.
  • User Interface: Functions to develop and maintain the user interface, providing an intuitive and user-friendly experience for interacting with the platform.
  • Documentation: Functions to document the project, including user guides, API documentation, and code documentation to facilitate usage and development.
  • Testing: Functions to create and execute test cases to ensure the reliability, functionality, and performance of the platform.
  • Deployment: Functions to deploy the application to web servers or cloud platforms, ensuring accessibility and scalability for users.

2.3 User Classes and Characteristics

The "vroomData (Auto Park Smart Explorer)" project caters to two primary user classes: administrators and analysts.

Administrators are granted extensive privileges within the platform. They have the authority to manage the overall system, including user accounts and data access. Administrators can create, edit, and delete user accounts, ensuring the smooth functioning of the platform. They also have the ability to invite other users to access the platform and can oversee a list of users with access permissions.

Analysts, on the other hand, are the users invited by administrators to utilize the platform for data analysis. While they cannot perform administrative tasks like creating or deleting accounts, analysts have the capability to explore and manipulate data within the confines of their access permissions. They can view and modify data related to the auto park statistics, enabling them to derive insights and make informed decisions.

Both administrators and analysts have the option to create personal accounts, providing them with access to the platform's functionalities. They can log in to their accounts securely and delete them if necessary.

2.4 Operating Environment

This application is designed to be used on a web browser, so it can be accessed from any device that has an internet connection and a web browser installed. The application is designed to be used on a desktop computer, but it can also be used on a mobile device, as it is responsive and it can be used on any screen size.

  • Data Accuracy: Ensuring the data retrieved and displayed is accurate and reliable.
  • Performance: Handling large datasets efficiently for quick response times.
  • Security: Implementing robust security measures for data protection.
  • Scalability: Accommodating future growth in data volume and user base.
  • Compatibility: Ensuring the application works across major web browsers and devices.

2.6 User Documentation

2.7 Assumptions and Dependencies

In order to function at its maximum potential, the "vroomData" web application needs to be connected to the internet, as it uses an external database to store the information related to the profiles of the users. The application also needs to be connected to the internet in order to be able to download the files needed.

No external hardware is required in order to use the "vroomData" web application, as it can be used on any device that has an internet connection and a web browser installed.

3. External Interface Requirements

3.1 User Interfaces

In order to have access to the "vroomData" web application, a user needs to have an account. The user can be redirected to create account by clicking on the “Sign Up” button right under the "Welcome!" message displayed on the home page of the application.

To create an account, the user needs to enter their e-mail, first name, last name, date of birth, password and they also need to confirm their password.

After the user has entered all the required information, they can click on the “Sign Up” button in order to create their account.

After the user has created their account, they can log in to the application by entering their e-mail and password by pressing "Log in!" on the home page of the application.

3.2 Hardware Interfaces

  • Servers: The backend services, including the REST/GraphQL API, run on robust server infrastructure capable of handling high volumes of requests and large datasets. These servers should be equipped with sufficient CPU, memory, and storage resources to ensure optimal performance.
  • User Devices: The application is designed to be accessed from a variety of user devices, including:
    • Desktops/Laptops: Modern web browsers (e.g., Chrome, Firefox, Edge, Safari) are supported.
    • Tablets and Smartphones: The responsive design ensures usability on mobile devices, providing a consistent experience across different screen sizes and resolutions.

3.3 Software Interfaces

    The APa application is built using a combination of HTML, CSS, PHP, and JavaScript to create a dynamic and responsive user interface.
  • HTML (HyperText Markup Language): Used for structuring the web pages and presenting content to the user.
  • CSS (Cascading Style Sheets): Employed for styling the web pages, ensuring a visually appealing and consistent design.
  • PHP (Hypertext Preprocessor): Utilized for server-side scripting, handling data retrieval, processing, and interactions with the REST/GraphQL API.
  • JavaScript: Used for client-side scripting, enabling dynamic content updates, interactive visualizations, and improved user experience through frameworks like D3.js for data visualization and Leaflet.js for map integration.

3.4 Communications Interfaces

  • REST API: The application communicates with the backend services using RESTful APIs. These APIs are designed to be stateless and use standard HTTP methods (GET, POST, PUT, DELETE) for data operations.
  • GraphQL API: An alternative communication interface that allows clients to request specific data structures, reducing over-fetching and under-fetching of data.
  • WebSockets: For real-time data updates and notifications, WebSockets may be used to maintain a persistent connection between the client and server.
  • HTTPS: All communications between the client and server are encrypted using HTTPS to ensure data security and integrity.

4. System Features

4.1 Authentification

4.1.1 Description and Priority

The authentification system is used to verify the identity of the user. The user needs to enter their e-mail and password in order to log in to the application. The user can also create an account by entering their e-mail, first name, last name, date of birth, password and they also need to confirm their password.

The authentification system is used in the following situations:

  • The user wants to log in to the application.
  • The user wants to create an account.

The authentification system is a very important feature of the vroomData web application, as it is the only way to verify the identity of the user and to make sure that the user is the only one who has access to the information related to the user's account.

4.1.2 Stimulus/Response Sequence List

The following stimulus/response sequence list describes the authentification system:

  1. The user enters their e-mail, first name, last name, date of birth, password and they also need to confirm their password in order to create an account.
  2. If all the information is correct, the user may continue by clicking on the “Sign Up” button.
  3. After all this, the user is redirected to the home page of the application.

If any of the information is incorrect, the user will be notified and an account will not be created.

4.1.3 Functional Requirements

The authentification system has the following functional requirements:

  • The user needs to enter their e-mail, first name, last name, date of birth, password and they also need to confirm their password in order to create an account.
  • The user needs to enter their e-mail and password in order to log in to the application.

4.2 Exporting a CSV/WebP/SVG file

4.2.1 Description and Priority

The exporting a CSV/WebP/SVG file system is used to export a CSV/WebP/SVG file with all the information related to the auto park information.

The exporting a CSV/WebP/SVG file system is a very important feature of the vroomData web application, as it allows the user to export a file containing all the information related to the auto park information such that a backup can be made, should the vroomData web application be unavailable.

4.2.2 Stimulus/Response Sequence List

The following stimulus/response sequence list describes the exporting a CSV/JSON file system:

  1. The user clicks on the “Export CSV/WebP/SVG” button.
  2. The user selects the format of the file that they want to export.
  3. The user may continue by clicking on the “Export” button.

4.2.3 Functional Requirements

  • User Authentication: Users should be able to create accounts, log in, and manage their profiles securely.
  • Data Retrieval: The application should allow users to retrieve automotive data based on various filters such as year, make, model, and type.
  • Data Visualization: Users should be able to view data in multiple formats, including bar charts, line graphs, pie charts, and geographic maps.
  • Export Functionality: Users should be able to export data and visualizations in CSV, WebP, and SVG formats.
  • Interactive Interface: The application should provide a responsive and interactive interface, allowing users to interact with the visualizations and maps dynamically.
  • Search and Filter: Users should have robust search and filter options to narrow down the data they are interested in.

5. Other Non Functional Requirements

5.1 Performance Requirements

  • Response Time: The application should have a fast response time, with data retrieval and visualization generation taking no more than 3 seconds for standard queries.
  • Scalability: The system should handle up to 10,000 concurrent users without performance degradation.
  • Data Volume: The backend should efficiently manage large datasets, capable of processing and serving data for over 10 million vehicle records.

5.2 Security Requirements

When a user creates an account, the application will send an email to the email address that the user provided, in order to verify that the email address is valid. The user will have to click on a link in the email in order to verify the email address and activate the account. The user will not be able to log in to the application until the email address is verified.[*]

The data kept by the application regarding the users' profiles is not shared with any third party. The data is only used by the application and the users that have access to the profiles.

Regarding the security of the application, the passwords of the users are stored in the database in an encrypted form, using an algorithm we will choose later. The passwords are not stored in plain text.
